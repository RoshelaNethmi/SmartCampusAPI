# Smart Campus Sensor & Room Management API ### 5COSC022W Client-Server Architectures — University of Westminster ## Overview A RESTful API built with JAX-RS (Jersey 2.32) on Apache Tomcat 9. Manages campus Rooms and Sensors with full CRUD operations, type filtering, sub-resource readings history, and enterprise error handling with custom Exception Mappers and Logging Filters. **Base URL:** `http://localhost:8080/SmartCampusAPI/api/v1` | Method | Endpoint | Description | |--------|----------|-------------| | GET | /api/v1 | API discovery + links | | GET | /api/v1/rooms | List all rooms | | POST | /api/v1/rooms | Create a room | | GET | /api/v1/rooms/{id} | Get one room | | DELETE | /api/v1/rooms/{id} | Delete room (blocked if has sensors) | | GET | /api/v1/sensors | List all sensors | | GET | /api/v1/sensors?type=X | Filter sensors by type | | POST | /api/v1/sensors | Register new sensor | | GET | /api/v1/sensors/{id}/readings | Get reading history | | POST | /api/v1/sensors/{id}/readings | Add new reading | --- ## How to Build and Run ### Prerequisites - Java JDK 8 or later - Apache NetBeans 12+ - Apache Tomcat 9.x - Maven (included with NetBeans) ### Steps 1. Clone this repository: ```bash git clone https://github.com/RoshelaNethmi/SmartCampusAPI.git ``` 2. Open NetBeans → File → Open Project → select the cloned folder 3. API is live at: http://localhost:8080/SmartCampusAPI/api/v1 --- ## Sample curl Commands ```bash # 1. API Discovery curl -X GET http://localhost:8080/SmartCampusAPI/api/v1 # 2. Instance variables are reset each time. This is why all data is stored in static fields inside DataStore.java — static fields belong to the class itself so they persist across all requests. ### Part 1 Q2 — HATEOAS HATEOAS means responses include hyperlinks to available actions. Clients discover URLs from responses instead of hardcoding them from documentation. This reduces coupling — if the server moves a resource, only the link in the response changes, not clients. ### Part 2 Q1 — IDs vs Full Objects Returning only IDs saves bandwidth but causes N+1 requests. Returning full objects costs more data but gives clients everything in one round trip, which is preferred for most cases. ### Part 2 Q2 — DELETE Idempotency Yes. First DELETE removes the room and returns 204. A second DELETE finds nothing and returns 404. The server state after both is identical — the room does not exist. ### Part 3 Q1 — @Consumes and Media Type Mismatch @Consumes(APPLICATION_JSON) tells JAX-RS to only accept requests with Content-Type: application/json. If a client sends text/plain JAX-RS automatically rejects it with 415 Unsupported Media Type before the method body even runs. ### Part 3 Q2 — @QueryParam vs Path Segment Query parameters are for filtering — they are optional modifiers on a collection. A path like /sensors/type/CO2 implies type is a resource identity not a filter. Omitting a query param naturally returns all results without any code changes. ### Part 4 Q1 — Sub-Resource Locator Pattern The pattern delegates a URL sub-path to a separate class. Without it SensorResource would contain all reading methods too, growing to hundreds of lines. Each class stays focused on one concern and is easier to test independently. ### Part 5 Q1 — 422 vs 404 for Missing Reference 404 means the URL does not identify a resource. 422 means the URL is valid but the JSON payload is semantically wrong. When roomId does not exist the URL /sensors is perfectly valid — the problem is inside the body, so 422 is more accurate. ### Part 5 Q2 — Stack Trace Security Risk Stack traces expose class names, method names, line numbers and library versions. Attackers use this to find known vulnerabilities. GlobalExceptionMapper logs the trace server-side only and returns a generic message to the client. ### Part 5 Q3 — Filters vs Manual Logging A filter applies automatically to every request. Manual Logger calls must be added to every method and are easy to forget when adding new endpoints. One filter enforces uniform logging with zero changes to resource classes.
